# Developer Landing API

Backend-сервис формы обратной связи для лендинга разработчика. Сервис валидирует и
сохраняет обращения, анализирует их с помощью OpenAI, отправляет два email-уведомления,
ограничивает частоту запросов и предоставляет защищённую агрегированную статистику.

По адресу `/` доступен адаптивный frontend-лендинг с формой, подключённой к API.
Служебный frontend для health-check и защищённых метрик доступен по `/monitoring`.

## Возможности

- `POST /api/contact` — приём обращения;
- `/` — адаптивный лендинг и форма обратной связи;
- `/monitoring` — простой интерфейс health-check и метрик;
- `GET /api/health` — проверка приложения и БД;
- `GET /api/metrics` — статистика без персональных данных;
- `/docs` и `/openapi.json` — Swagger UI и OpenAPI 3.0;
- строгая валидация тела запроса и запрет неизвестных полей;
- rate limiting с ответом `429` и заголовком `Retry-After`;
- AI-классификация, анализ тональности, приоритет и черновик ответа;
- graceful fallback при отсутствии ключа, таймауте или ошибке OpenAI;
- два HTML-письма: владельцу и пользователю;
- request ID, ротация файловых логов и единый формат ошибок;
- SQLite, SQLAlchemy и миграции Alembic;
- автоматические тесты без реальных обращений к OpenAI и SMTP.

## Стек

- Python 3.13, Flask 3;
- Flask-Smorest, Marshmallow, OpenAPI;
- Flask-SQLAlchemy, Flask-Migrate, SQLite;
- Flask-Limiter, Flask-CORS;
- OpenAI Responses API и Structured Outputs;
- стандартные `smtplib` и `email`;
- pytest, Ruff, Gunicorn, Docker.

Для AI по умолчанию выбран `gpt-5.6-luna`: модель предназначена для небольших
стоимостно-чувствительных задач и поддерживает Responses API и Structured Outputs.
Модель можно заменить через `OPENAI_MODEL`.

## Архитектура

```text
app/
├── api/           # HTTP-контроллеры и Blueprints
├── schemas/       # входные и выходные схемы Marshmallow
├── services/      # бизнес-сценарии, AI и email
├── repositories/  # работа с SQLAlchemy
├── models/        # модели базы данных
├── middleware/    # request ID и журналирование запросов
├── errors/        # глобальные обработчики ошибок
├── prompts/       # версионируемые AI-промпты
└── templates/     # HTML-шаблоны писем
```

Полный поток обращения:

```text
HTTP → validation → rate limit → ContactService → repository
                                      ├─→ OpenAI / fallback
                                      └─→ owner + customer email
```

Контроллер отвечает только за HTTP-контракт. `ContactService` координирует сценарий,
репозиторий изолирует хранение, а внешние провайдеры находятся в отдельных сервисах.

## Локальный запуск

Требуется Python 3.13+.

```bash
git clone <repository-url>
cd internet_labratory
python3.13 -m venv .venv
source .venv/bin/activate
pip install -r requirements-dev.txt
cp .env.example .env
flask --app run:app db upgrade
flask --app run:app run --debug
```

API будет доступен на `http://127.0.0.1:5000`, Swagger — на
`http://127.0.0.1:5000/docs`.

Для запуска production-сервера:

```bash
gunicorn --bind 0.0.0.0:8000 --workers 1 --threads 4 run:app
```

## Переменные окружения

Скопируйте `.env.example` в `.env`. Файл `.env` не должен попадать в Git.

| Переменная | Назначение |
|---|---|
| `SECRET_KEY` | соль для хеширования IP и секрет Flask |
| `DATABASE_URL` | строка подключения SQLAlchemy |
| `CORS_ORIGINS` | разрешённые origin через запятую |
| `TRUST_PROXY_HOPS` | число доверенных reverse-proxy hops; локально `0` |
| `OPENAI_API_KEY` | ключ OpenAI; пустое значение включает fallback |
| `OPENAI_MODEL` | модель AI |
| `AI_TIMEOUT_SECONDS` | таймаут AI-запроса |
| `EMAIL_MODE` | `console` для разработки или `smtp` |
| `SMTP_*` | параметры SMTP-сервера |
| `EMAIL_FROM` | адрес отправителя |
| `OWNER_EMAIL` | адрес владельца сайта |
| `CONTACT_RATE_LIMIT` | лимит формы, например `5 per 15 minutes` |
| `RATELIMIT_STORAGE_URI` | backend Flask-Limiter |
| `METRICS_API_KEY` | ключ доступа к `/api/metrics` |
| `LOG_FILE` | путь к файлу логов |

Для production задайте случайные длинные значения `SECRET_KEY` и `METRICS_API_KEY`,
укажите точный домен в `CORS_ORIGINS` и не используйте `*`.

## API

### POST `/api/contact`

```bash
curl -i http://127.0.0.1:5000/api/contact \
  -H 'Content-Type: application/json' \
  -d '{
    "name": "Иван Петров",
    "phone": "+7 (777) 123-45-67",
    "email": "ivan@example.com",
    "comment": "Хочу обсудить разработку веб-сервиса."
  }'
```

Успешный ответ, `201 Created`:

```json
{
  "id": 1,
  "status": "accepted",
  "message": "Обращение принято",
  "ai_processed": false,
  "ai": {
    "category": "other",
    "sentiment": "neutral",
    "priority": "normal",
    "reply": "Спасибо за обращение. Я получил ваше сообщение и свяжусь с вами в ближайшее время."
  }
}
```

`ai_processed: false` означает, что применён безопасный fallback. Обращение при этом
сохранено и продолжает обрабатываться.

Правила валидации:

- `name`: строка длиной 2–100 символов;
- `phone`: 7–15 цифр, допускаются `+`, пробелы, скобки и дефисы;
- `email`: корректный адрес длиной до 254 символов;
- `comment`: строка длиной 10–3000 символов;
- неизвестные поля отклоняются.

### GET `/api/health`

```bash
curl http://127.0.0.1:5000/api/health
```

Показывает доступность БД и режимы AI/email, но не раскрывает ключи и персональные данные.

### GET `/api/metrics`

```bash
curl http://127.0.0.1:5000/api/metrics \
  -H 'X-API-Key: your-metrics-key'
```

Возвращает количество обращений, обращения за сутки, число AI-fallback, ошибки email и
распределение по категориям. Endpoint защищён сравнением API-ключа с постоянным временем.

### Ошибки и HTTP-статусы

| Статус | Ситуация |
|---|---|
| `201` | обращение принято |
| `400` | некорректный HTTP/JSON-запрос |
| `401` | неверный ключ метрик |
| `404` | endpoint не существует |
| `422` | ошибка валидации полей |
| `429` | превышен rate limit |
| `500` | неожиданная внутренняя ошибка |
| `503` | метрики не настроены |

Единый формат ошибки:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request validation failed",
    "request_id": "c7eec7d0-...",
    "details": {}
  }
}
```

`X-Request-ID` возвращается также в заголовке ответа и позволяет найти запрос в логах.

## AI-интеграция и fallback

AI получает только комментарий и возвращает структурированный объект:

```json
{
  "category": "project_request",
  "sentiment": "positive",
  "priority": "normal",
  "reply_draft": "Спасибо за обращение..."
}
```

Допустимые значения ограничены Pydantic-схемой. Системный промпт хранится в
`app/prompts/contact_analysis.py`; он объявляет комментарий недоверенными данными,
запрещает исполнять инструкции из него и запрещает обещать сроки или цены.

Fallback срабатывает, если ключ отсутствует, провайдер недоступен, превышен таймаут или
ответ не соответствует схеме. Тогда сохраняются нейтральная классификация и шаблонный
ответ. Ошибка журналируется, но endpoint продолжает работу и возвращает `201`.

## Email

`EMAIL_MODE=console` не отправляет письма, а безопасно пишет факт формирования письма в
лог без его содержимого. `EMAIL_MODE=smtp` отправляет HTML-письмо владельцу и копию
пользователю. Пользовательские значения экранируются Jinja автоматически.

Сбой SMTP не удаляет уже сохранённое обращение. Статусы двух писем хранятся отдельно и
учитываются в метриках.

## Хранение, rate limiting и логи

- обращения и агрегируемая статистика хранятся в SQLite;
- миграции находятся в `migrations/`;
- сырой IP не сохраняется — только SHA-256 от `SECRET_KEY + IP`;
- request log не содержит тело формы, email, телефон или комментарий;
- логи ротируются: 5 файлов максимум по 1 МБ;
- локально rate limit хранится в памяти процесса;
- `TRUST_PROXY_HOPS` включается только за доверенным reverse proxy;
- для нескольких production-инстансов следует задать Redis-compatible storage URI.

## Docker

```bash
cp .env.example .env
docker compose up --build
```

API откроется на `http://127.0.0.1:8000`. Контейнер автоматически применяет миграции,
запускается не от root, а база и логи хранятся в Docker volumes.

## Деплой

В репозитории есть `render.yaml`. После создания сервиса в Render задайте секретные
переменные из `.env.example`. Для сохранения SQLite между перезапусками нужен persistent
disk; альтернативно замените `DATABASE_URL` на управляемую SQL-базу с соответствующим
драйвером. После деплоя проверьте `/api/health`, `/docs`, реальную отправку SMTP и fallback
с временно удалённым `OPENAI_API_KEY`.

## Что сделано с помощью AI

AI использовался как ассистент при подготовке каркаса, вариантов архитектуры, тестовых
сценариев и первичной документации. Вручную были проверены и скорректированы:

- границы слоёв и порядок сохранения обращения;
- схема Structured Outputs и prompt-injection ограничения;
- отсутствие персональных данных в request log и metrics;
- поведение при ошибках AI и SMTP;
- HTTP-статусы, rate limiting и тесты;
- миграции и запуск из чистого окружения.

Примеры запросов к AI-ассистенту во время разработки:

```text
Предложи слоистую структуру Flask API для формы обратной связи.
Перечисли сценарии отказа OpenAI и SMTP, которые необходимо покрыть тестами.
Проверь, что контроллер не содержит бизнес-логику и не раскрывает внутренние ошибки.
```

AI-результаты не принимались без проверки: код запускался, миграции применялись, а
критические fallback-сценарии покрыты автоматическими тестами.

## Ограничения и развитие

- memory rate-limit рассчитан на один процесс; для горизонтального масштабирования нужен Redis;
- SQLite подходит для тестового и небольшого сервиса; при росте нагрузки — PostgreSQL;
- email отправляется синхронно; следующим шагом будет очередь задач и повторные попытки;
- `/api/metrics` — прикладная статистика, а не Prometheus-формат.
